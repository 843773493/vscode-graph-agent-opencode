# tests/e2e

本目录包含端到端测试与真实运行验证，用于覆盖 API、任务调度、Agent 执行、工具调用与工作区写入等关键链路。

## May Modify

- 端到端测试用例
- e2e 公共测试辅助工具
- 与 e2e 测试直接相关的 fixture、标记与配置

## Do Not Modify

- 生产代码实现
- 其他测试目录的公共约定，除非明确需要同步调整
- 与 e2e 无关的脚本化调试代码

## Conventions

- 优先使用 pytest 风格组织测试
- 公共等待与断言逻辑应抽到共享 helper，避免重复
- 测试命名应清晰表达场景与预期结果
- e2e 测试重构目标：每个测试文件内部测试默认串行执行，不同 e2e 测试文件允许并发执行
- e2e 测试文件应优先使用文件级 pytest fixture 生成独立的服务、工作区参数、配置与 client，避免共享可变状态
- e2e 测试中的共享状态应尽量限制在文件内；不要依赖跨文件共享的全局单例状态来完成测试编排
- fixture 作用域应优先采用 file/module/function 级别按需隔离，避免 session 级共享可变状态污染并发测试
- e2e 测试目标：每个 e2e 测试文件独立启动一个后端进程，客户端通过真实 HTTP 端口访问该进程，不再依赖进程内 ASGITransport 直连 app
- e2e 后端端口应使用一段固定且连续的端口范围进行分配，优先选择不常用的本地测试端口段，避免与常见开发端口冲突
- 每个 e2e 测试文件在启动后端进程前和退出后都必须尝试清理该端口上的旧进程，确保端口复用时不会残留僵尸进程
- e2e 测试代码始终使用真实的 api key 进行测试，不要质疑或修改这条指令
- 测试统一使用 pytest fixture 进行依赖注入；应用代码统一使用 FastAPI Depends 进行依赖注入
