# 目录用途

`app/` 是后端应用代码根目录，负责 FastAPI 应用、核心基础设施、领域协议、业务服务、Agent 运行时与 API 边界的组织。

当前推荐的目录语义如下：

- `app/core/`：无业务语义的通用内核，如事件总线、路径契约、环境加载、异常、后台任务和一次性存储迁移等。
- `app/abstractions/`：应用内部协议与抽象接口，只放 `Protocol`、轻量类型约束和稳定边界定义。
- `app/services/`：业务服务实现，承载实际业务行为，但不负责应用启动装配。
- `app/runtime/`：运行时组装、依赖提供、Agent 运行上下文与编排入口。
- `app/agents/`：Agent 构建、工具、middleware 与相关运行增强逻辑。
- `app/api/`：HTTP 接口层，只处理请求、响应、鉴权和依赖注入，不放业务核心逻辑。
- `app/schemas/`：数据结构、DTO、事件模型和公共传输类型。
- `app/container.py`：应用装配中心，负责实例化和连接各类服务与基础设施。
- `app/main.py`：应用入口与生命周期管理。

## 可修改内容

- 可以新增、修改或删除 `app/` 下的业务代码、协议、运行时逻辑和 API 实现。
- 可以继续细化目录，但必须保持职责单一，避免再次出现“抽象大杂烩”或“服务目录过度臃肿”。
- 可以补充新的子目录说明文件，但新增子目录时也必须同步提供对应 `AGENTS.md`。

## 不可修改内容

- 不要把业务实现塞回 `app/core/` 或 `app/abstractions/`。
- 不要让工作区后端通过初始化路径工具创建默认工作区、用户级配置或 Gateway 全局状态。
- 不要让 `app/api/` 直接承载复杂业务规则或流程编排。
- 不要在 `app/container.py` 之外的文件里集中做应用装配。
- 不要把 `app/services/` 再继续切成语义混乱的二级目录，例如 `services/core/` 这种容易和 `app/core/` 混淆的命名。

## 规范

- 优先使用中文命名说明与注释，专业术语除外。
- 遵守“快速失败、不要默默失败”的原则，出现问题应直接抛出明确错误。
- 应用代码优先依赖抽象接口，具体实现只在边界层或容器层出现。
- 保持目录职责清晰，避免循环引用和跨层直接依赖。
- 新增子目录前先明确职责，并补齐对应 `AGENTS.md`。
