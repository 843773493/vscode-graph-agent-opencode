# 重构计划：src/webview UI 框架选型与重构方案

## 当前问题

`src/webview/html.js` 1231 行 HTML/CSS 字符串硬编码 + 旧版 Webview 原生 DOM 操作，无组件化、无类型、难以维护。

## 调研结论

对比了 cline（React + Vite）和 kilocode（SolidJS + esbuild）两个参考实现后，以下是针对**本项目实际约束**的深度分析。

---

## 方案 A：cline 路线 — React 18 + Vite + 纯 CSS

### 架构

```
src/webview-ui/           # 新增独立目录（参考 cline webview-ui/）
├── index.html
├── vite.config.ts        # Vite 构建 + HMR
├── src/
│   ├── main.tsx          # ReactDOM.createRoot
│   ├── App.tsx           # 根组件
│   ├── context/
│   │   └── vscode.tsx    # acquireVsCodeApi + postMessage 封装
│   ├── components/       # 按现有 UI 拆分为 React 组件
│   └── styles/
│       └── index.css     # 现有 CSS 平移，不引入 Tailwind
```

### 关键依赖

| 包 | 体积(gzip) | 作用 |
|---|---|---|
| react + react-dom | ~42KB | 运行时 |
| vite | dev | 构建 + HMR |
| @vitejs/plugin-react-swc | dev | SWC 编译 JSX |

### 优势

1. **HMR 开发体验最优** — 修改组件后 webview 自动热更新，不丢状态，无需重启 VS Code（参考 cline `getHMRHtmlContent()`）
2. **生态最广** — React 组件库、StackOverflow 答案、AI 代码生成质量都是最好的
3. **团队适配成本最低** — 大多数前端开发者熟悉 React
4. **Vite 生态成熟** — CSS @import、assets、环境变量等都开箱即用

### 劣势

1. **虚拟 DOM 开销** — SSE 流式消息场景下，每秒多次 `setState` 触发虚拟 DOM diff，长会话性能下降
2. **Bundle 体积较大** — React 运行时 42KB+，总 bundle 预计 80-150KB
3. **Hooks 心智负担** — `useEffect` 依赖数组、`useCallback` 闭包陷阱、stale closure 等问题

### 对本项目的适配成本

- **CSS**：保持现有纯 CSS 方案，不引入 Tailwind（避免重写 600+ 行样式）
- **通信**：保持现有 `postMessage` 协议，不引入 gRPC
- **类型**：可逐步迁移到 TypeScript，Vite 原生支持

## 方案 B：kilocode 路线 — SolidJS + Vite + 纯 CSS

### 架构

```
src/webview-ui/
├── index.html
├── vite.config.ts        # Vite + vite-plugin-solid
├── src/
│   ├── main.tsx          # render(() => <App />, root)
│   ├── App.tsx           # 根组件
│   ├── context/
│   │   └── vscode.tsx    # SolidJS context 封装
│   ├── components/       # SolidJS 组件
│   └── styles/
│       └── index.css     # 现有 CSS 平移
```

### 关键依赖

| 包 | 体积(gzip) | 作用 |
|---|---|---|
| solid-js | ~7KB | 运行时（极小的响应式系统） |
| vite | dev | 构建 + HMR |
| vite-plugin-solid | dev | SolidJS JSX 编译 |

### 优势

1. **性能最佳** — 无虚拟 DOM，细粒度响应式。SSE 流式消息场景下，仅更新变更的文本节点，长会话无性能衰减
2. **Bundle 体积极小** — SolidJS 运行时仅 ~7KB，总 bundle 预计 30-60KB
3. **响应式语义直观** — `createSignal` / `createEffect` 更接近原生思维，没有 hooks 规则
4. **JSX 编译优化** — 编译时展开响应式，运行时无 diff 开销

### 劣势

1. **生态较小** — 第三方组件库少，大部分 UI 需自研
2. **招聘/维护成本** — 会 SolidJS 的前端开发者远少于 React
3. **JSX 陷阱** — SolidJS 的 JSX 与 React 有微妙差异（如 `{signal()}` 必须调用、组件只执行一次），容易踩坑
4. **调试体验** — 由于编译时转换，错误堆栈有时难以理解

### 对本项目的适配成本

- **CSS**：与方案 A 相同，保持现有纯 CSS
- **通信**：与方案 A 相同
- **状态迁移**：现有原生 DOM 代码中大量 `innerHTML` 更新，迁移到 SolidJS 的细粒度响应式需要重新设计状态结构

## 深度分析：SSE 流式性能

这是本项目最关键的场景。当前旧版 Webview 处理 SSE 事件时：

```js
// 当前实现：直接操作 DOM，性能最好但代码最乱
outputEl.innerHTML += `<div>${chunk}</div>`;

// React 方案：虚拟 DOM diff
setMessages(prev => [...prev, newMessage]); // 每次 setState 触发全量 diff

// SolidJS 方案：细粒度更新
setMessages(prev => [...prev, newMessage]); // 编译时优化，仅追加 DOM 节点
```

React 在消息列表超过 50 条、每秒 3-5 次更新时会出现明显卡顿（尤其是在低端机器上）。SolidJS 在这种情况下性能接近原生 DOM 操作。

**缓解措施**：React 方案可以通过 `react-virtuoso` 虚拟列表（cline 已使用）来解决长列表性能问题，但引入额外依赖。

## 建议

如果**团队有 React 经验**且**对招聘/长期维护敏感** → **选 React + Vite**。SSE 性能问题用虚拟列表缓解。

如果**团队能接受小众技术**且**SSE 流式性能是硬性指标** → **选 SolidJS + Vite**。
