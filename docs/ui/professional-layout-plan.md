# VS Code Graph Agent 专业级 UI 排版方案

## 目标

当前 UI 的核心问题不是“组件不够多”，而是**布局责任和内容责任混在一起**，导致历史栏、聊天区、输入区在不同宽度下容易出现挤压、固定宽度失效、内容区不可用等问题。

本方案的目标是把 UI 重构为一套**专业、稳健、可扩展**的排版体系：

1. **主聊天区始终优先可用**
2. **历史栏在窄宽度下自动折叠为抽屉**
3. **布局由统一的壳组件管理，不让内容组件控制宽度**
4. **宽屏可并排，窄屏可覆盖，极窄时自动隐藏辅助面板**
5. **整体风格尽量贴近 VS Code 原生 Webview / Copilot Chat 体验**

### 设计总原则

要做出 Copilot 风格，重点不是“长得像”，而是遵守下面这些 UI 纪律：

1. **内容与布局彻底分离**：`HistoryPanel`、`ChatPanel`、`Composer` 只负责渲染内容，不负责自己占多宽。
2. **宽度必须范围化**：所有面板宽度都应使用 `clamp()`、`minmax()` 或容器驱动规则，避免固定像素宽度主导布局。
3. **主任务区优先**：聊天区和输入区始终优先显示，辅助区必须可折叠、可隐藏，不能反向压制主流程。
4. **所有溢出都要可控**：路径、ID、badge、长文本、代码块都必须有截断、折行或独立滚动策略。
5. **视觉层次统一**：颜色、边框、间距、字号必须遵循 VS Code 主题变量和一致的密度节奏。

## 现状问题

### 1. 固定宽度容易挤压主内容

当前历史栏如果使用固定宽度，在 VS Code Sidebar 场景中非常容易把聊天区压缩到不可用。

### 2. 组件职责不够纯粹

`HistoryPanel`、`ChatPanel`、`Composer` 这类内容组件不应该知道自己的布局策略，但目前容易出现“组件自己决定宽度”的倾向。

### 3. 需要更强的宽度适配能力

VS Code Webview 的容器尺寸并不稳定：
- 侧边栏宽度小
- 编辑器区宽度大
- 用户还可能拖拽面板尺寸

因此需要布局能根据容器宽度自动切换模式，而不是依赖固定像素值。

### 4. 右侧遮挡与完全遮挡问题仍然可能出现

即便采用响应式布局，如果存在下面这些情况，仍然会发生右侧内容被遮挡或完全遮挡：

- 抽屉层使用 `position: absolute` / `transform` 覆盖主内容，但关闭时未彻底退出布局语义
- 长文本没有统一做 `overflow` / `ellipsis` / `word-break` 约束
- `flex` / `grid` 子项没有设置 `min-width: 0`，导致内容把父容器撑爆
- 某些区域仍是“内容驱动宽度”，没有被统一布局壳接管

所以本项目的真正目标不是“看上去自适应”，而是**让任何辅助面板都无法挤压或覆盖主聊天区的可视性**。

## 推荐架构

### 一、把布局与内容彻底分离

#### 内容组件
只负责渲染内容，不负责排版策略：
- `HistoryPanel`
- `ChatPanel`
- `Composer`
- `Toolbar`

#### 布局壳组件
统一负责空间分配和模式切换：
- `AppShell`
- `ResponsiveWorkspaceLayout`
- `PanelModeResolver`

---

### 二、采用“自适应 + 抽屉 + 分栏”三态布局

#### 模式 1：Docked（并排停靠）
适用于宽屏场景。

- 历史栏位于左侧
- 聊天区占据剩余空间
- 适合编辑器区域或宽侧栏

#### 模式 2：Drawer（抽屉覆盖）
适用于中等宽度或侧边栏场景。

- 历史栏不再占固定列
- 点击历史按钮后，以浮层方式覆盖聊天区
- 关闭后聊天区恢复全宽

#### 模式 3：Hidden（自动隐藏辅助区）
适用于极窄宽度。

- 仅显示聊天区和输入区
- 历史面板默认隐藏
- 防止主任务区被压缩到不可用

## 布局策略

### 1. 布局优先级

排版策略按以下优先级决策：

1. 先保证聊天区最小可用宽度
2. 再决定是否显示历史栏
3. 历史栏显示时，优先使用抽屉而不是硬挤
4. 只有在容器足够宽时才使用并排布局

### 1.1 Copilot 风格的空间纪律

为避免出界、遮挡和奇怪的覆盖，布局需要遵守以下纪律：

- 根容器必须 `overflow: hidden`，避免子元素横向溢出扩散到页面外。
- 中间内容区必须 `flex: 1 1 0%` / `min-height: 0` / `min-width: 0`，避免内容撑爆容器。
- 底部输入区必须放在正常流里，不应通过绝对定位叠压消息列表。
- 抽屉、弹层、菜单必须有明确 z-index 分层与遮罩语义，关闭时要彻底退出交互层。

---

### 2. 宽度规则建议

建议使用“范围”而不是“固定值”：

- 历史栏最小宽度：`220px`
- 推荐宽度：`clamp(240px, 24vw, 320px)`
- 聊天区最小宽度：`minmax(0, 1fr)`
- 当容器不足时自动切换为抽屉模式

---

### 3. 布局实现建议

优先考虑以下技术：

#### 方案 A：CSS Grid
适合左右分栏的结构控制。
- 逻辑清晰
- 适合停靠布局
- 便于定义最小列宽

#### 方案 B：Container Query
适合根据 webview 实际容器宽度切换模式。
- 比 `window.innerWidth` 更准确
- 非常适合 VS Code Webview

#### 方案 C：状态驱动的布局模式切换
由 `AppShell` 统一决定当前布局模式：
- `docked`
- `drawer`
- `hidden`

#### 方案 D：三态自动收缩策略

根据容器宽度自动切换：

- `> 960px`：`docked`，历史栏并排停靠
- `600px ~ 960px`：`drawer`，历史栏默认收起，需要时覆盖展开
- `< 600px`：`hidden`，直接隐藏历史栏，只保留主聊天区

这套策略比“固定显示 + 固定宽度”更接近 Copilot 的实际体验。

## 组件拆分建议

### 1. `AppShell`
负责整体骨架。

职责：
- 渲染顶部工具栏
- 渲染主内容区
- 渲染底部输入区
- 控制历史栏显示模式

### 2. `ResponsiveWorkspaceLayout`
负责主区域的空间分配。

职责：
- 判断当前容器宽度
- 决定历史栏是并排、抽屉还是隐藏
- 管理主聊天区的最小宽度

### 3. `HistoryPanel`
只负责列表渲染。

职责：
- 展示会话列表
- 高亮当前会话
- 通知上层切换会话

不应该负责：
- 固定宽度
- 位置摆放
- 折叠逻辑

### 4. `ChatPanel`
只负责消息流。

职责：
- 渲染 turn
- 渲染 trace/output
- 展示空态

### 5. `Composer`
只负责输入与发送。

职责：
- 文本输入
- 发送消息
- 快捷键交互

不应该负责：
- 布局位置
- 面板尺寸

### 6. `ContextChip` / `FollowupActions` / `StepList`

建议在后续把 Copilot 风格的关键交互拆成独立组件：

- `ContextChip`：展示引用文件、工作区上下文、选中内容
- `StepList`：把工具调用和 trace 结果转成语义化步骤，而不是原始 JSON
- `FollowupActions`：在回复结束后展示“继续解释 / 生成测试 / 重构”等后续动作

这些组件应该作为 `ChatPanel` 和 `Composer` 的子层，不应反过来参与布局决策。

## 交互建议

### 1. 历史栏按钮行为

历史按钮不只是“开/关”，而应该根据当前模式执行不同动作：

- 在宽屏：切换显隐
- 在窄屏：打开抽屉
- 在极窄屏：直接隐藏并仅保留入口

### 2. 主聊天区永远优先

任何布局切换都不能影响：
- 消息可见性
- 输入框可操作性
- 发送按钮可见性

### 3. 自动隐藏规则必须明确

历史栏和辅助面板不是“手动关一次就完事”，而应该有明确的自动隐藏规则：

- 宽屏默认显示，但不允许压缩主聊天区到不可读
- 中等宽度默认收起为抽屉，避免永久占位
- 极窄宽度直接隐藏，只保留主任务流和入口按钮
- 抽屉打开时必须有遮罩层，避免发生“看起来关了但其实还挡着”的状态

### 3. 可选增强：拖拽分栏

如果后续需要更专业的 IDE 感，可以增加：
- 历史栏宽度拖拽
- 用户自定义默认宽度
- 宽度记忆

但这一步建议放在第二阶段做。

## 样式原则

### 1. 不要在内容组件里写死布局宽度

例如下面的做法应避免：
- `width: 288px`
- `margin-left: 288px`
- `left: 288px`

这些写法容易导致：
- 窄宽度崩坏
- 未来扩展困难
- 响应式切换复杂

### 2. 使用语义化样式变量

建议将样式抽象为：
- `--layout-sidebar-min-width`
- `--layout-sidebar-max-width`
- `--layout-content-min-width`
- `--layout-gap`

建议进一步补充这些变量，形成一套更像 Copilot 的设计系统：

- `--toolbar-height`
- `--panel-header-height`
- `--composer-min-height`
- `--space-xs / sm / md / lg`
- `--radius-sm / md / lg`
- `--elevation-overlay`
- `--z-drawer`
- `--z-overlay`

### 3. 统一 VS Code 主题变量

尽量使用 VS Code Webview 暴露的主题 CSS 变量，避免硬编码颜色。

### 4. 文本处理统一规范

所有容易溢出的文本都必须统一处理：

- 单行标题、路径、ID：`overflow: hidden; text-overflow: ellipsis; white-space: nowrap;`
- 长 Markdown / JSON / trace：独立滚动或 `word-break: break-word; overflow-wrap: anywhere;`
- 代码块：单独工具条 + 单独滚动容器，避免撑爆卡片宽度

## 实施步骤

### Phase 1：抽离布局责任

- 将历史栏宽度控制从 `HistoryPanel` 中移除
- 在 `AppShell` 或布局壳中集中管理宽度和切换逻辑
- 先保证并排模式和抽屉模式能切换

### Phase 2：加入容器宽度感知

- 使用容器宽度而不是浏览器窗口宽度
- 根据宽度自动切换 `docked / drawer / hidden`

### Phase 3：优化视觉层级

- 简化顶部工具栏
- 弱化非核心按钮权重
- 让聊天区成为视觉中心

### Phase 4：增强专业交互

- 历史栏拖拽调整宽度
- 输入区自动增高
- 面板状态记忆
- 更自然的过渡动画

### Phase 5：Copilot 风格化收尾

- 统一消息卡片、输出卡片、输入卡片的视觉节奏
- 补充 hover-only 的次级操作按钮
- 降低工具栏按钮占比，让内容区成为视觉中心
- 清理所有会导致出界/覆盖的固定尺寸和绝对定位逻辑

---

## 验收标准

当这套布局方案落地后，应满足以下标准：

1. 在 VS Code Sidebar 中，聊天区仍然可正常使用
2. 历史栏不会把主内容挤到不可用
3. 宽屏下可以并排显示，窄屏下自动切换为抽屉
4. `HistoryPanel`、`ChatPanel`、`Composer` 都不再依赖固定宽度
5. 主题切换时样式仍然自然
6. 用户拖拽面板尺寸时不会出现布局崩坏
7. 长路径、ID、代码块、JSON 输出不会把右侧内容挤出可视区
8. 历史栏 / 抽屉不会在关闭后仍遮挡主聊天区
9. 辅助区域在窄宽度下会自动收缩或隐藏，不会“硬占位”

---

## 结论

如果目标是“最专业”，推荐采用：

> **内容组件纯渲染 + 布局壳统一调度 + 容器宽度感知 + 历史栏抽屉化 + 主聊天区优先**

这套方案比固定宽度、硬拼左右栏的方式更适合 VS Code Webview，也更利于后续功能扩展。
